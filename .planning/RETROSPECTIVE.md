# 里程碑复盘

## Milestone: v1.0 — generateGrammar 重构

**Shipped:** 2026-05-05  
**Phases:** 3  
**Plans:** 13  
**Tasks:** 38

### What Was Built（完成内容）

- 建立 LoPS 科研脚本重构流程和目录契约。
- 完成 `generateGrammar.py` 默认分支的深度分析、调用模块重构、数据迁移、运行入口和验证入口。
- 将正式核心模块设计为不依赖旧项目代码、旧项目数据目录和旧格式输出结构。
- 将旧格式转换隔离到 `script/generate_grammar/legacy_adapter.py`，只用于新旧结果比对。
- 完成 `generate_grammar` 顶层算法优化，并用 34 被试全量输出和过程级 trace 验证结果一致。

### What Worked（有效做法）

- 先分析再讨论再计划，能显著减少科研脚本重构中的误删和误改风险。
- 每个 phase 对应一项完整重构，比把一次重构拆成多个 phase 更符合用户审查节奏。
- 把数据迁移到 `data/` 后，正式代码的路径边界更清晰，也更容易验证是否仍依赖旧项目。
- 旧格式转换放在验证脚本中，而不是放入核心模块，避免了兼容逻辑反向约束新架构。
- 过程一致性测试能发现“最终结果一致但中间路径可能偏离”的风险，适合科研算法重构。

### What Was Inefficient（低效点）

- 一开始对 quick task 的标准 `SUMMARY.md` 文件名约定不够严格，导致归档前 `audit-open` 误报悬挂任务。
- Phase 2 的早期输出设计仍保留了旧格式兼容思路，后续又通过 quick task 和 Phase 3 清理，说明规则应该更早写清。
- 顶层算法优化阶段开始时仍偏结构设计，之后才收敛到算法函数和过程变量；以后算法优化应先列顶层算法和关键中间量。
- 独立 `*-VALIDATION.md` Nyquist 标准文件没有同步生成，虽然已有 `VERIFICATION.md` 和测试覆盖，但格式上仍不够统一。

### Patterns Established（沉淀模式）

- 重构 phase 必须包含脚本分析、讨论、设计、执行和验证的完整闭环。
- 目标脚本调用到的本地模块默认纳入同轮重构和模块级行为测试。
- 正式核心模块不做旧格式兼容；旧格式比对通过独立验证适配器完成。
- 运行数据、测试数据、验证输出和基准文件必须放在 `data/`，不放入 `.planning/`。
- 算法优化从顶层流程和数据流开始，先确定主循环和关键中间变量，再重写底层函数。

### Key Lessons（经验）

- 对科研脚本来说，“新旧输出一致”还不够；关键算法应记录并比较中间过程变量。
- 如果旧代码没有显式随机过程，不要为了测试便利加入额外随机控制。
- 命名和数据结构要优先表达当前项目语义，不要被旧输出字段牵引。
- GSD 元数据也需要收口验证；归档前应固定运行 `gsd-sdk query audit-open`。

### Cost Observations（成本观察）

- 最大成本来自理解旧脚本行为、数据含义和验证边界，而不是代码搬迁本身。
- 过程级验证脚本虽然是临时代码，但能显著提升重构结论可信度。
- 对后续重构而言，越早明确“哪些旧分支不参与本轮重构”，计划和验证成本越可控。
