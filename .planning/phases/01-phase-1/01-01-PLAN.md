# Plan 01-01: 建立最小 run intake 模板

<objective>
创建 `.planning/runs/` 的最小可用骨架，让每一轮科研脚本重构都能用中文 Markdown 记录目标脚本、运行环境和数据来源。本计划只做 intake 入口契约，不提前设计完整 analysis、proposal、verification 模板或复杂状态系统。
</objective>

<context>
本计划落实 `.planning/phases/01-phase-1/01-CONTEXT.md` 中的 D-01、D-02、D-03、D-04、D-07、D-08、D-09、D-10、D-11。

- D-01: 每轮重构的入口信息以中文 Markdown 模板为主，不在 Phase 1 强制引入 JSON/YAML manifest。
- D-02: Markdown 模板需要适合人工补充背景，字段固定但允许占位和说明性文字。
- D-03: 目标脚本路径、运行环境、数据来源三项是进入后续分析的最低必填信息。
- D-04: 运行命令、必要权限、预期输出可以先占位，但模板必须显式标记为“待补充”。
- D-07: 每轮重构使用独立目录记录状态和材料，推荐根路径为 `.planning/runs/<run-id>/`。
- D-08: Phase 1 只建立最小可用骨架：`.planning/runs/` 目录约定和一个 intake Markdown 模板，不提前设计完整状态系统。
- D-09: 不把科研重构轮次直接塞进 `.planning/phases/...`。
- D-10: 每轮目录名采用 `YYYY-MM-DD-short-name`。
- D-11: 短名使用小写 ASCII、数字和连字符。
</context>

<threat_model>
## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| intake 模板诱导用户粘贴密钥、token 或密码 | HIGH | 模板必须明确写出“不要记录 API keys、密码、token、私有凭据”。 |
| 外部脚本路径或数据路径被误当作本仓库内安全路径 | MEDIUM | 模板必须要求记录路径来源、是否需要额外权限，以及是否位于项目外部。 |
| 可选字段被下游误认为已确认 | MEDIUM | 运行命令、权限、预期输出等可选字段必须默认标记为“待补充”。 |
</threat_model>

创建 `.planning/runs/README.md`，使用中文说明：
