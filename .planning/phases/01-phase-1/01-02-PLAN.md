# Plan 01-02: 补齐项目级 README 目录职责说明

<objective>
创建或修正项目级 `README.md`，用中文说明 LoPS 的用途、每个关键目录的职责、`.planning/runs/` 的最小使用方式，以及第一轮真实重构应该从填写 intake 开始。
</objective>

<context>
本计划落实 `.planning/phases/01-phase-1/01-CONTEXT.md` 中的 D-05、D-06，并引用 Plan 01-01 创建的 `.planning/runs/README.md` 和 `.planning/runs/INTAKE-TEMPLATE.md`。

- D-05: Phase 1 应创建或更新一个简洁的项目级 README，说明 `src/LoPS`、`script`、`data`、`docs`、`.planning` 的职责。
- D-06: 不在 Phase 1 为每个目录都创建目录内 README，避免早期文档过散。
- D-08: Phase 1 必须保持简单，只交付启动第一轮重构所需的最小骨架。
</context>

<threat_model>
## Threat Model

| Threat | Severity | Mitigation |
|--------|----------|------------|
| README 让用户误以为 Phase 1 会执行具体重构 | MEDIUM | README 必须明确写出 Phase 1 只建立入口骨架，具体脚本分析从后续 run intake 开始。 |
| README 目录异常导致覆盖用户内容 | HIGH | 执行前必须检查 `README.md` 当前是文件还是目录；若目录非空，停止并询问用户，不得删除内容。 |
| README 暗示可以记录敏感凭据 | HIGH | README 必须明确写出不要把密钥、token、密码写入 intake 或规划文档。 |
</threat_model>

<action>
创建或更新根目录 `README.md`，使用中文，至少包含这些章节和内容：

1. `# LoPS`
2. `## 项目目标`：说明 LoPS 用于多轮科研脚本重构，不是每轮新建 GSD project。
3. `## 目录职责`：逐条说明：
   - `src/LoPS/`：可复用正式模块。
   - `src/LoPS/temp/`：仅用于验证阶段的临时旧实现副本，验证后清理。
   - `script/`：运行新模块或验证流程的脚本入口。
   - `data/`：确认需要纳入项目的数据。
   - `docs/`：项目说明、提示和人工文档。
   - `.planning/phases/`：GSD 阶段计划和上下文。
   - `.planning/runs/`：多轮科研脚本重构实例。
4. `## 开始一轮重构`：说明新建 `.planning/runs/YYYY-MM-DD-short-name/intake.md`，从 `.planning/runs/INTAKE-TEMPLATE.md` 复制内容。
5. `## 最低必填信息`：列出目标脚本路径、运行环境、数据来源。
6. `## 安全提醒`：写出不要记录 API keys、密码、token 或私有凭据。
7. `## 当前阶段边界`：写出 Phase 1 只建立最小入口骨架，不分析具体脚本、不执行重构。

不要创建每个目录内的 README，不要引入 JSON/YAML manifest。
