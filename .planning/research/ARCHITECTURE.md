# 架构研究

**领域：** 科研 Python 脚本重构与一致性验证  
**研究日期：** 2026-05-03  
**置信度：** MEDIUM

## 标准架构

### 系统概览

```text
用户提供目标脚本、环境、数据
        |
        v
任务接收与原始行为记录
        |
        v
分析层：依赖、输入、输出、执行流程、随机过程
        |
        v
方案层：模块边界、接口、数据路径、验证计划
        |
        v
实现层：src/LoPS 模块 + script 运行入口 + data 数据组织
        |
        v
验证层：旧实现输出 vs 新实现输出
        |
        v
记录层：结果、限制、后续事项
```

### 组件职责

| 组件 | 职责 | 典型实现 |
|------|------|----------|
| 任务接收记录 | 保存目标脚本、环境、数据来源和预期运行方式 | Markdown 记录或后续结构化 manifest |
| 原始分析器 | 理解脚本入口、依赖、输入输出和随机性 | 人工分析为主，后续可沉淀辅助函数 |
| 模块边界设计 | 判断功能是否进入 `src/LoPS`，定义接口 | 方案文档和计划阶段产物 |
| 重构模块 | 承载可复用科研计算逻辑 | `src/LoPS/<domain>.py` 或子包 |
| 运行入口 | 用整理后的数据调用新模块 | `script/<name>.py` |
| 验证工具 | 运行旧版本与新版本并比较输出 | pytest 或独立验证脚本 |
| 临时旧代码区 | 保存注入随机种子的旧实现副本 | `src/LoPS/temp`，验证后清理 |

## 推荐项目结构

```text
src/
└── LoPS/
    ├── __init__.py              # 包入口，后续需要时添加
    ├── temp/                    # 仅用于临时旧实现验证
    └── <module>.py              # 每轮确认后新增或扩展的重构模块
script/
└── <run_feature>.py             # 面向用户的运行入口
data/
└── <feature>/                   # 整理后的输入数据和必要样例
docs/
└── prompt.md                    # 初始项目意图
.planning/
├── PROJECT.md
├── REQUIREMENTS.md
├── ROADMAP.md
└── research/
```

### 结构理由

- `src/LoPS` 只放可复用逻辑，避免把一次性运行流程混入库模块。
- `script` 放可直接执行的脚本，便于用户复现实验。
- `data` 放整理后的项目数据，但外部原始数据只在确认需要迁移时复制。
- `src/LoPS/temp` 是验证辅助区，不是长期代码归档。

## 架构模式

### 模式 1：先封装 I/O，再提纯计算

**内容：** 先明确原脚本读取什么、写出什么，再把核心计算函数从路径和命令行状态中分离出来。  
**适用：** 原脚本大量使用相对路径、全局变量或直接读写文件。  
**取舍：** 前期分析更慢，但能减少重构后结果漂移。

### 模式 2：旧新双入口验证

**内容：** 保留旧实现运行方式，同时提供新实现运行入口，用同一输入比较输出。  
**适用：** 任何结果敏感的科研脚本。  
**取舍：** 需要维护验证脚本，但这是证明一致性的核心。

### 模式 3：随机性显式化

**内容：** 新模块通过参数或配置接收 seed；旧脚本无 seed 时，在临时副本中加入统一 seed。  
**适用：** Monte Carlo、模拟、随机初始化或采样流程。  
**取舍：** 不能随意改原始脚本，只能在临时验证副本中做最小注入。

## 数据流

```text
外部目标脚本 + 外部数据
        |
        v
原始运行输出基准
        |
        v
整理后的 data 输入
        |
        v
script 运行新模块
        |
        v
新实现输出
        |
        v
比较器：完全一致或有记录的数值容差
```

## 主要反模式

| 反模式 | 问题 | 替代 |
|--------|------|------|
| 直接把整个旧脚本搬进 `src/LoPS` | 只是移动文件，没有形成模块边界 | 先识别计算核心和 I/O 边界 |
| 先改路径再理解流程 | 路径行为常常影响输出 | 先记录原始运行方式和输出 |
| 验证只看是否无报错 | 无法证明科研结果未变 | 比较旧新输出 |
| 长期保留 `temp` 旧代码 | 污染正式仓库结构 | 验证后删除，并记录验证结果 |
