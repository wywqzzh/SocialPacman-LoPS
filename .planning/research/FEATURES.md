# 功能研究

**领域：** 科研 Python 脚本重构与一致性验证  
**研究日期：** 2026-05-03  
**置信度：** MEDIUM

## 功能版图

### 基础能力

| 功能 | 为什么需要 | 复杂度 | 说明 |
|------|------------|--------|------|
| 目标脚本接收 | 每轮工作都依赖用户指定外部脚本 | LOW | 需要记录路径、运行命令、环境和数据 |
| 原始行为分析 | 不理解旧逻辑就无法安全重构 | MEDIUM | 包括输入、输出、流程、依赖和数据关系 |
| 模块化必要性评估 | 并非所有脚本都值得抽象成库模块 | MEDIUM | 依据复杂度、复用价值、边界清晰度判断 |
| 重构方案确认 | 用户明确要求确认后再改代码 | LOW | 方案要包含文件拆分、接口、数据路径和验证方式 |
| 模块化实现 | 将确认后的逻辑落入 `src/LoPS` | HIGH | 需要兼顾可读性和行为一致 |
| 数据和运行脚本整理 | 用户需要能运行迁移后的模型或功能 | MEDIUM | 数据进入 `data`，入口进入 `script` |
| 一致性验证 | 科研结果不能因重构改变 | HIGH | 比较旧实现和新实现的输出 |
| 随机过程处理 | 未固定随机种子会导致无法公平对比 | HIGH | 临时复制旧代码并注入统一种子 |

### 可形成差异化的能力

| 功能 | 价值 | 复杂度 | 说明 |
|------|------|--------|------|
| 每轮重构报告 | 让用户清楚知道改了什么、如何验证 | MEDIUM | 可在后续阶段加入 |
| 自动生成验证脚本模板 | 降低每轮重复劳动 | MEDIUM | 先建立手工模式，再抽象模板 |
| 输出差异诊断 | 当结果不一致时定位差异来源 | HIGH | 可后置到验证能力稳定之后 |
| 环境快照记录 | 提升科研复现性 | MEDIUM | 记录 Python、包版本、关键命令 |

### 反功能

| 功能 | 表面吸引力 | 问题 | 替代方案 |
|------|------------|------|----------|
| 自动重构所有外部脚本 | 看起来省事 | 范围不可控，容易误改未理解代码 | 用户逐轮指定目标 |
| 未确认方案直接执行 | 速度更快 | 违反用户工作流，风险高 | 先方案，后确认，再执行 |
| 只做 smoke test | 成本低 | 无法证明科研结果一致 | 默认做旧新输出对比 |
| 强制清洗所有数据路径 | 项目结构统一 | 可能破坏原始路径假设 | 先记录原路径，再设计显式路径参数 |

## 依赖关系

```text
目标脚本接收
    -> 原始行为分析
        -> 模块化必要性评估
            -> 重构方案确认
                -> 模块化实现
                    -> 数据和运行脚本整理
                        -> 一致性验证
                            -> 完成记录
```

### 依赖说明

- 原始行为分析必须早于方案确认，否则方案只能基于猜测。
- 方案确认必须早于模块化实现，这是用户明确设置的门。
- 数据和运行脚本整理必须早于一致性验证，否则新实现没有稳定入口。
- 随机过程处理是验证阶段的分支：只有原始脚本存在未固定随机过程时启用。

## MVP 定义

### v1 必须具备

- [ ] 接收并记录每轮目标脚本、环境和数据来源。
- [ ] 完整分析原始脚本功能、输入输出和依赖。
- [ ] 产出等待用户确认的重构方案。
- [ ] 根据确认方案创建 `src/LoPS` 模块、`script` 入口和必要 `data` 组织。
- [ ] 运行旧实现与新实现，对比输出一致性。
- [ ] 对未固定随机过程执行临时旧代码注种验证，并清理临时代码。

### 验证后再加入

- [ ] 自动化验证脚本模板。
- [ ] 环境快照和依赖记录。
- [ ] 差异诊断报告。

### v2 以后考虑

- [ ] 批量扫描外部项目并建议重构顺序。
- [ ] 图形化任务管理界面。
- [ ] 多语言科研脚本支持。
