# 风险研究

**领域：** 科研 Python 脚本重构与一致性验证  
**研究日期：** 2026-05-03  
**置信度：** MEDIUM

## 关键风险

### 风险 1：只重构结构，丢失结果一致性

**问题：** 新代码更清晰，但输出与旧脚本不一致。  
**原因：** 没有先记录旧输出，或者比较时使用了不同输入、路径、seed 或依赖版本。  
**避免方式：** 每轮先建立原始运行基准，再实现新入口，最后使用同一输入做对比。  
**预警信号：** 方案中只写模块拆分，不写验证方式。  
**处理阶段：** Phase 2 和 Phase 5。

### 风险 2：把未理解的脚本直接模块化

**问题：** 旧脚本隐藏的全局状态、路径假设和副作用被带入新模块。  
**原因：** 在分析输入输出之前就开始移动代码。  
**避免方式：** 原始行为分析必须覆盖入口、依赖、读写文件、参数、随机过程和外部状态。  
**预警信号：** 新模块接口仍然依赖当前工作目录或隐式全局变量。  
**处理阶段：** Phase 2 和 Phase 3。

### 风险 3：随机过程不可复现

**问题：** 旧新输出无法比较，差异无法归因。  
**原因：** 原始脚本存在随机采样或初始化，但没有固定 seed。  
**避免方式：** 在 `src/LoPS/temp` 中复制旧代码并最小化注入统一 seed，新实现也显式接收同一 seed。  
**预警信号：** 输出每次运行都不同，但方案没有随机性处理。  
**处理阶段：** Phase 5。

### 风险 4：数据迁移改变语义

**问题：** 数据移入 `data` 后，原脚本相对路径、文件名排序或隐式默认文件变化。  
**原因：** 只关注文件位置，没有记录原始数据读取规则。  
**避免方式：** 先记录原始数据来源和读取顺序，再设计显式路径参数。  
**预警信号：** 迁移数据后旧脚本无法复现原输出。  
**处理阶段：** Phase 2 和 Phase 4。

### 风险 5：确认门被绕过

**问题：** 代码已经修改，但用户还没有认可方案。  
**原因：** 把分析、规划、执行混成一个动作。  
**避免方式：** GSD 阶段必须先形成方案，再进入执行。  
**预警信号：** Phase 3 之前出现业务代码改动。  
**处理阶段：** Phase 3。

## 技术债模式

| 捷径 | 短期收益 | 长期成本 | 何时可接受 |
|------|----------|----------|------------|
| 手工比较少量打印输出 | 快 | 覆盖不足，难以复现 | 只能作为初步 smoke check |
| 把数据路径硬编码进模块 | 省参数设计 | 复用困难，运行目录敏感 | 不应作为正式模块设计 |
| 保留大量临时旧代码 | 方便回看 | 仓库污染，用户混淆 | 验证期间短暂存在 |
| 忽略依赖版本 | 少记录 | 输出差异难以解释 | 仅在纯标准库脚本中可接受 |

## “看起来完成但其实没有”的检查清单

- [ ] **原始脚本分析：** 是否记录了输入、输出、依赖和执行命令？
- [ ] **方案确认：** 是否在修改实现前得到用户同意？
- [ ] **数据整理：** 是否能说明每个迁移数据文件来自哪里？
- [ ] **运行脚本：** 是否从 `data` 读取并调用 `src/LoPS` 模块？
- [ ] **一致性验证：** 是否对比了同一输入下的旧输出和新输出？
- [ ] **随机过程：** 若存在随机性，是否固定了公平对比所需的 seed？
- [ ] **临时代码：** `src/LoPS/temp` 验证后是否清理？

## 风险到阶段映射

| 风险 | 预防阶段 | 验证方式 |
|------|----------|----------|
| 结果不一致 | Phase 2, Phase 5 | 保存旧输出并比较新输出 |
| 未理解脚本就模块化 | Phase 2 | 分析记录覆盖入口、I/O、依赖 |
| 随机不可复现 | Phase 5 | 旧临时副本和新实现使用同一 seed |
| 数据语义改变 | Phase 4 | 新运行脚本用整理后数据复现输出 |
| 绕过确认门 | Phase 3 | 方案确认后才进入实现 |
