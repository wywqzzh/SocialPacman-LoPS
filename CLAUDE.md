# LoPS 项目协作说明

## 项目目标

LoPS 用于系统性重构外部科研脚本。每轮工作都应先理解目标脚本的原始行为，再形成重构方案并等待用户确认，最后才执行模块化、数据整理、运行脚本编写和一致性验证。

## 工作语言

- 与用户交流使用中文。
- Markdown 文档使用中文。
- 代码标识符、命令、路径、配置键名按技术语境使用英文。

## 目录约定

- `src/LoPS/`：可复用的正式模块。
- `src/LoPS/temp/`：仅用于验证阶段的临时旧实现副本，验证后必须清理。
- `script/`：运行新模块或验证流程的脚本入口。
- `data/`：确认需要纳入项目的数据。
- `docs/`：项目说明、提示和人工文档。
- `.planning/`：GSD 规划、需求、路线图、状态和研究资料。

## 强制流程

1. 用户提供目标脚本、运行环境和数据来源。
2. 先阅读目标脚本及其依赖，记录原始功能、输入输出、执行流程、依赖文件和数据关系。
3. 判断是否需要在 `src/LoPS` 下创建独立模块。
4. 在修改实现代码前，提出重构方案并等待用户确认。
5. 确认后实施重构，必要时整理数据到 `data` 并在 `script` 中创建运行入口。
6. 使用相同输入运行原始实现和新实现。
7. 若存在随机过程：
   - 原始代码已有 seed：沿用或显式记录。
   - 原始代码没有 seed：复制临时旧代码到 `src/LoPS/temp`，只做最小 seed 注入。
8. 对比旧新输出，默认要求完全一致；若使用数值容差，必须记录原因和容差。
9. 验证通过后删除 `src/LoPS/temp` 中本轮临时代码。

## 质量要求

- 不要在方案确认前修改业务实现代码。
- 不要为了代码整洁牺牲科研结果一致性。
- 不要把一次性运行流程混进可复用模块。
- 不要保留未清理的临时旧代码。
- 每轮完成时说明运行方式、验证方式和一致性结论。
