# LoPS 项目协作说明

## 项目目标

LoPS 用于系统性重构外部科研脚本。每轮工作都应先理解目标脚本的原始行为，再形成重构方案并等待用户确认，最后才执行模块化、数据整理、运行脚本编写和一致性验证。

## 工作语言

- 与用户交流使用中文。
- Markdown 文档使用中文。
- 代码标识符、命令、路径、配置键名按技术语境使用英文。

## 目录约定

- `src/LoPS/`：可复用的正式模块。
- `src/LoPS/temp/`：仅用于验证阶段的临时旧实现副本，验证后必须清理。
- `script/`：数据分析主流程入口按执行顺序以序号文件放在根目录，例如 `script/01_mat_to_raw_subject_data.py`；`script/pacman_video/` 保持视频入口。
- `data/`：当前脚本和测试使用的数据，包括需要纳入项目的输入、脚本输出和验证输出。
- `docs/`：项目说明、提示和人工文档。
- `.planning/`：GSD 规划、需求、路线图、状态和研究资料。

## 强制流程

1. 用户提供目标脚本、运行环境和数据来源。
2. 先阅读目标脚本及其依赖，记录原始功能、输入输出、执行流程、依赖文件和数据关系。
3. 判断是否需要在 `src/LoPS` 下创建独立模块。
4. 在修改实现代码前，提出重构方案并等待用户确认。
5. 确认后实施重构，必要时整理数据到 `data` 并在 `script` 中创建运行入口。
6. 使用相同输入运行原始实现和新实现。
7. 若存在随机过程：
   - 原始代码已有 seed：沿用或显式记录。
   - 原始代码没有 seed：复制临时旧代码到 `src/LoPS/temp`，只做最小 seed 注入。
8. 对比旧新输出，默认要求完全一致；若使用数值容差，必须记录原因和容差。
9. 验证通过后删除 `src/LoPS/temp` 中本轮临时代码。

## 质量要求

- 不要在方案确认前修改业务实现代码。
- 不要为了代码整洁牺牲科研结果一致性。
- 不要把一次性运行流程混进可复用模块。
- 不要保留未清理的临时旧代码。
- 不要把脚本输入、脚本输出、测试产物或验证产物放入 `.planning`；这些数据必须放在 `data` 下的对应子目录中，`.planning` 只保存计划、讨论、分析和结论文档。
- `src/LoPS` 中不得保存旧项目或其它项目的数据目录、代码目录等绝对路径；运行脚本可以为本仓库 `data/` 下的固定目录设置默认参数，测试和其它调用方也应显式构造配置。
- 新版本正式代码必须完全不依赖旧版本任何内容，包括旧版本代码、旧版本数据目录和旧版本数据文件；所有运行、测试、验证需要使用的数据都必须复制或整理到当前 LoPS 仓库的 `data/` 下，所有输出也必须写入当前 LoPS 仓库。
- 新版本设计阶段默认不考虑旧版本兼容性，包括不为了兼容旧版本输出数据格式而污染新模块的数据结构、接口和内部流程；新设计应优先表达当前项目中更清晰、更高内聚低耦合的模型。
- 若需要进行新旧结果比对，可以专门编写验证脚本或独立适配模块，把新版本输出转换成旧版本输出格式后再比较；此类转换逻辑必须与正式核心模块隔离，不能反向约束新版本架构。
- 所有新增或修改的代码都必须加入详细中文注释；注释的目标是说明功能、解释处理过程、标明关键数据含义和边界条件，而不是反复与旧版本代码做比对。
- 每个函数和类都必须在定义体开头使用三引号 docstring 说明该函数或类的功能、输入输出语义和关键约束。
- 在函数或方法内部，应在重点逻辑、难点逻辑、数据形态转换和容易误解的边界条件处添加中文中间注释，帮助读者理解实现过程；避免为显而易见的赋值或语法写空泛注释。
- 每轮完成时说明运行方式、验证方式和一致性结论。
