# 多对象 tile 抽帧问题与解决方法

## 问题

旧的单人数据中，Pacman 通常会在一个 tile 上停留较稳定的一段时间，因此可以用固定间隔抽帧近似得到“每个 tile 一个行为点”的数据。

现在的数据不同。任务中同时存在两个 Pacman 和两个 ghost，分析目标也不是单个玩家的轨迹，而是两个玩家在同一局面下的合作和竞争。因此，抽帧不能围绕某一个玩家单独进行，也不能简单地每隔固定帧数抽一帧。

从当前数据的连续同坐标片段来看，移动节奏已经明显不同于旧数据：

- 旧数据中，Pacman 常见做法是每 25 帧抽一帧，因为单个 tile 行为大致能被 25 帧窗口覆盖。
- 当前数据中，两个 Pacman 的连续同坐标片段主要集中在 12 帧和 13 帧。合并 p1 和 p2 后，12 帧片段占 43.61%，13 帧片段占 42.01%，两者合计占 85.62%。
- 两个 ghost 的连续同坐标片段也主要集中在 12 帧和 13 帧。合并 ghost1 和 ghost2 后，12 帧片段占 35.84%，13 帧片段占 35.38%，两者合计占 71.22%。
- ghost 仍然存在较多 25 帧片段，合并后 25 帧片段占 7.17%。这说明不同对象的停留节奏并不完全一致。

这些数值说明，继续使用旧的 25 帧固定抽帧会错过大量 12/13 帧尺度上的位置变化；但简单改成每 12 或 13 帧抽一次也不够，因为 Pacman 和 ghost 的切换并不同步。

核心困难是：多个对象的 tile 坐标不是完全同步切换的。

例如，在一次局面变化中，p1 可能先进入新 tile，随后 ghost1 在几帧后进入新 tile，p2 或 ghost2 又在更后面的几帧进入新 tile。逐帧数据会把这些变化记录在不同帧上，但从 tile 行为分析角度看，它们更像是同一次整体局面变化的异步展开。

如果只要任意对象位置变化就保留该帧，会在一个短时间窗口内留下多条候选帧。这会导致：

1. 一个实际 tile 行为步被拆成多个伪步骤。
2. 相邻 tile 行之间经常出现“某个 Pacman 没动，但其它对象动了”的记录。
3. 后续计算动作方向时，Pacman 会产生大量空方向或难解释的状态。
4. 行为序列比真实 tile 决策序列更密，影响合作和竞争关系的分析。

但如果简单地只保留每个短窗口的最后一帧，也会有风险：

1. 某个对象可能在被删除的中间帧里经过了必要的相邻 tile。
2. 压缩后相邻两行之间可能出现跨 tile 跳跃。
3. 死亡、刷新等短暂状态可能只在中间帧出现，被误删后会丢失关键事件。

因此，需要一种方法既能压缩异步切换造成的冗余候选帧，又能保证压缩后的序列仍然是空间连续、状态可解释的。

## 整体状态切换区间

“整体状态切换区间”是为了解决这个问题而定义的分析概念。它不是游戏程序中显式记录的事件，也不是 Pacman 规则里的状态。

这里的“整体状态”指多个对象共同组成的局面，例如：

- p1 的位置和状态
- p2 的位置和状态
- ghost1 的位置
- ghost2 的位置

如果游戏在 tile 层面完全同步，那么这些对象应该在同一个逻辑步里一起完成更新。此时一行 tile 数据就可以代表一次完整的整体局面变化。

但真实逐帧记录中，不同对象的 tile 坐标切换可能相差几帧。于是，同一次整体局面变化会在原始记录中展开成一个短时间窗口。这个短时间窗口就是“整体状态切换区间”。

换句话说，整体状态切换区间表示：一次多对象联合局面变化在逐帧数据中陆续完成的时间范围。

当前思路中，可以用一个经验帧长来定义这个区间。例如当窗口长度设为 13 帧时：

- 起始帧到起始帧后第 12 帧属于同一个整体状态切换区间。
- 起始帧后第 13 帧及之后的候选帧视为下一个区间。

这个区间的作用是给压缩候选帧提供边界：只有在同一个整体状态切换区间内，才尝试删除中间候选帧。

## 解决方法

解决方法分为两步。

### 第一步：生成联合状态变化候选帧

第一步不直接得到最终 tile 行为序列，而是先生成候选帧。

处理方式是：把多个对象的位置和关键状态组合成一个联合状态。每个 trial 的第一帧保留；之后只要当前帧的联合状态相对上一帧发生变化，就保留为候选帧。

联合状态至少应包含：

- p1 位置
- p1 状态
- p2 位置
- p2 状态
- ghost1 位置
- ghost2 位置

Pacman 状态需要参与比较，因为死亡或刷新可能发生在位置不变的帧上。如果只比较位置，这类关键状态变化会被漏掉。

第一步的作用是去掉完全重复的逐帧记录，但它只能生成候选帧，不能直接作为最终结果。

### 只做第一步的问题

假设第一步生成的候选帧序列如下：

```text
0, 14, 26, 27, 48, 51, 52, 73, 76, 77, ...
```

其中 26 和 27、48 和 51 和 52、73 和 76 和 77 这类短时间内连续出现的候选帧，通常对应的是不同对象在同一个整体状态切换区间内异步完成 tile 更新。

如果把这些候选帧全部保留，就会把一次整体局面变化拆成多个行为点。

但如果直接只保留区间最后一帧，也可能破坏连续性。例如上一条保留行到最后一帧之间，某个对象可能已经移动了两个 tile。如果删除中间行，压缩后的序列就不再表示真实的一步移动。

因此，需要第二步对候选帧进行有条件压缩。

### 第二步：压缩异步切换区间

第二步在候选帧基础上处理整体状态切换区间。

基本原则是：在同一个整体状态切换区间内，可以尝试删除中间候选帧，但删除后必须保证上一条保留行能够直接连续到下一条候选行。

更直观地说，可以把压缩过程理解成下面的伪代码：

```text
输入：候选帧序列 candidates
参数：整体状态切换区间长度 window，例如 13 帧

keep = [第一条候选帧]
previous_kept = 第一条候选帧
interval_start = 第一条候选帧的 frame_id

从第二条候选帧开始，依次把当前候选帧记为 current：

    next = current 后面的下一条候选帧

    如果 current 已经不在 interval_start 定义的窗口内：
        把 interval_start 更新为 current 的 frame_id

    如果 next 不存在：
        保留 current
        结束

    如果 next 已经超出当前整体状态切换区间：
        保留 current
        previous_kept = current
        interval_start = next 的 frame_id
        继续处理 next

    否则，next 仍在当前整体状态切换区间内：
        尝试删除 current

        如果 previous_kept 可以直接连续到 next
        并且 current 不携带独有关键状态：
            删除 current
            继续处理 next

        否则：
            保留 current
            previous_kept = current
            interval_start = next 的 frame_id
            继续处理 next
```

这个伪代码里的三个对象需要特别区分：

- `previous_kept`：上一条已经确定保留的帧。
- `current`：当前正在判断是否可以删除的候选帧。
- `next`：如果删除 `current`，将会直接接到 `previous_kept` 后面的下一条候选帧。

因此，删除 `current` 前真正要检查的是：`previous_kept -> next` 是否仍然合理，而不是只检查 `current -> next`。

这里的连续性是对联合状态中的每个对象分别检查：

- 位置不变，视为连续。
- 移动到地图上的合法相邻 tile，视为连续。
- 其它情况视为不连续。

也就是说，压缩不是简单地“每个区间只保留最后一帧”，而是在空间连续性和关键状态保护的约束下尽量删除冗余中间帧。

## 关键状态保护

除了位置连续性，还需要保护短暂但重要的状态变化。

例如 Pacman 死亡或刷新时，状态可能只在很短的时间内出现。如果某一候选帧的 Pacman 状态同时不同于上一条保留帧和下一条候选帧，说明它可能携带独有状态。这种帧不能因为位置可以连续而被删除。

因此，删除中间候选帧需要同时满足两个条件：

1. 删除后所有对象的位置仍然连续。
2. 被删除帧不携带前后都没有的关键状态。

## 为什么不固定间隔抽帧

固定间隔抽帧无法稳定处理双人任务。

如果分别按玩家抽帧，会得到两套不同的时间轴，无法直接描述同一时刻下两个玩家和 ghost 的关系。

如果按统一固定间隔抽帧，又会依赖起始相位。一旦抽到整体状态切换区间的中间位置，就可能得到半更新状态：某些对象已经进入新 tile，另一些对象还在旧 tile。

因此，固定间隔抽帧不适合当前问题。更合适的方法是先找联合状态变化候选帧，再压缩异步切换区间。

## 为什么不补帧

这个问题的目标是从逐帧记录中选择能代表 tile 行为序列的观测帧，而不是人为插入新的 tile 行。

如果在抽帧阶段补帧，会让每一行到底是真实观测帧还是人工构造帧变得不清楚。对于后续合作和竞争分析来说，保留真实观测帧的来源更重要。

因此，坐标修正应在前置步骤完成；抽帧阶段只负责选择和压缩候选帧，不额外补帧。

## 方法特点

这个方法的核心特点是：

1. 用联合状态而不是单个玩家状态作为抽帧单位。
2. 用整体状态切换区间解释多对象异步换 tile 的短时间窗口。
3. 用连续性检查避免压缩后出现跨 tile 跳跃。
4. 用关键状态保护避免删除死亡、刷新等短暂事件。
5. 压缩后再重新计算动作方向，保证动作字段对应最终序列。

最终得到的序列不是逐帧数据，也不是固定间隔采样数据，而是经过联合状态变化检测和异步切换压缩后的 tile 级行为序列。
