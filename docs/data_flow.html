Pacman 数据处理流程说明

本文档说明当前 LoPS 仓库中的 Pacman 数据处理流程。主分析流程按照 script/ 根目录中编号脚本的执行顺序组织；视频流程保留在 script/pacman_video/ 下，作为独立渲染链路。

当前默认数据根目录为 data/。主分析流程从 data/00_raw_mat_data 开始，逐阶段写入带编号的数据目录。

主分析流程

主分析流程只考虑两只鬼的数据。每个阶段尽量读取上一阶段输出，生成自己负责的新增字段或结构； 已经标准化后的字段会在后续阶段一致保留，避免重复解释和重复转换。

01. raw mat 到 raw subject data

脚本 script/01_mat_to_raw_subject_data.py

输入数据data/00_raw_mat_data 下的原始 Pacman .mat session 目录。
输出数据data/01_raw_subject_data 下每个 session 一个 *_raw_subject_data.pkl。

目的

将原始 MATLAB 数据整理为 Python 后续流程可直接读取的单被试逐 trial 表格。 该阶段保留原始记录中的位置、方向、模式、水奖励等基础字段，为 frame data 构建提供完整来源。

处理方式

1. 发现 session 并并行调度

脚本扫描 data/00_raw_mat_data 下的 session 目录。若命令行传入 session 白名单， 只处理指定 session；否则处理全部目录。每个 session 是一个独立任务，可以通过 --workers 并行。

2. 读取 MATLAB 原始结构

每个 session 内读取原始 .mat 数据，把 MATLAB table 中的逐帧记录转为 pandas 可处理的数据表。该阶段不做行为分析，只负责把原始采集字段搬到当前仓库的数据格式中。

3. 保留 raw 级基础字段

输出中保留后续 frame 构建需要的原始列，例如 DayTrial、Step、 Map、pacMan_1/2、ghost 坐标和状态、方向、帧编号、水奖励字段等。

4. 写出 session 级 pickle

每个 session 写成一个 *_raw_subject_data.pkl。文件仍然接近原始采集形态， 下游会在 frame 阶段再做去重、排序、过滤和坐标列表解析。

02. raw subject data 到 frame data

脚本 script/02_raw_subject_data_to_frame_data.py

输入数据data/01_raw_subject_data。
输出数据data/02_frame_data，可选 CSV 输出到 data/02_frame_data_csv。

目的

将逐 trial 原始记录转换为逐 frame 的基础数据表。该阶段生成统一的 frame_id，并且只保留两只鬼 trial；四只鬼 trial 会被过滤掉。

处理方式

1. 每帧去重

输入 raw subject data 可能存在同一 DayTrial-Step 多行记录。脚本按 DayTrial 和 Step 分组，并取每组首行，保证输出中每一行对应唯一 frame。

2. 过滤四鬼 trial

当前主分析流程只考虑两只鬼。脚本在 frame data 生成阶段按整局判断并过滤四鬼 trial， 避免后续地图常量和 utility 计算遇到当前流程不支持的位置或策略维度。

3. 数字顺序排序

输出前先按 DayTrial 的数字段和 Step 排序，而不是直接用字符串排序。 这样 2-1 会排在 10-1 前面，保证 frame_id 稳定。

4. 构造 frame 字段

Pacman 和 ghost 的格点位置会被整理成 tuple[int, int]；Map 字符串会被解析出 beans 和 energizers 坐标列表；渲染仍需使用的像素坐标、方向、帧编号和水奖励字段继续保留。

5. 生成 frame_id 并写出

排序完成后生成整数 frame_id，替代旧流程里的 Unnamed: 0。 输出文件名为 *_frame_data.pkl；只有传入 --write-csv 时才额外写 CSV。

03. frame data 预处理

脚本 script/03_frame_data_preprocess.py

输入数据data/02_frame_data。
输出数据data/03_preprocessed_frame_data。

目的

生成后续分析使用的标准 frame 表。该阶段是字段清理和标准化入口， 将 raw frame 中的分析必要字段整理成更小、更稳定的数据形态。

处理方式

1. 读取 raw frame 并校验必要字段

输入必须包含 DayTrial、Step、pacmanPos、 ghost1Pos、ghost2Pos、ifscared1/2、beans 和 energizers。缺少任一字段会直接报错。

2. 标准化 id 和 trial 字段

frame_id 被强制转为 int64；DayTrial 转为字符串； game_id 从 DayTrial 中去掉第二段 trial round 得到。例如 1-2-031222-401-03-Dec-2022 会生成 1-031222-401-03-Dec-2022。

3. 标准化坐标和列表

pacmanPos、ghost1Pos、ghost2Pos 被转为 tuple[int, int]。beans 和 energizers 被转为 list[tuple[int, int]]，缺失或空值按空列表处理。

4. 标准化 scared 状态

ifscared1 和 ifscared2 转为 int8； 缺失值用 -1 表示不存在或不可用状态。

5. 收敛为分析字段集合

输出只保留 frame_id、DayTrial、game_id、Step、 pacmanPos、两只鬼位置、两只鬼 scared 状态、beans 和 energizers。 action_dir 和 available_dir 在 tile 修正后再生成。

04. tile data 预处理

脚本 script/04_human_tile_data_preprocess.py

输入数据data/03_preprocessed_frame_data，迷宫邻接表 data/constant_data/adjacent_map_fmri.csv。
输出数据data/04_tile_data 和 data/04_corrected_tile_data。

目的

将逐 frame 数据转为 tile 级行为数据，并修正相邻位置缺失造成的路径断点。 该阶段生成动作字段，是后续 utility、拟合和特征计算的行为基础。

处理方式

1. 抽样生成 tile data

对每个 DayTrial 按 sample_rate 抽样，默认每 25 帧取一行。 每个 trial 的最后一帧会被强制保留，即使它不在固定采样间隔上。

2. 删除无效 tunnel 外坐标

Pacman 位于 (-1, 18) 或 (31, 18) 的记录会被删除。 这些位置是 tunnel 外的中间标记，不属于后续邻接表稳定覆盖的地图格点。

3. 修正路径连续性

corrected tile data 会检查相邻抽样点的 pacmanPos 是否连续。 若两个点之间缺少中间格，脚本会根据邻接路径插入缺失行，保证后续每一行到下一行都是一步动作。 左右 tunnel 的 (0, 18) 与 (30, 18) 按环形相邻处理。

4. 修正 ghost 特殊位置

ghost 位于家门口特殊坐标 (14, 20)、(15, 20)、(16, 20) 时， 会映射到门外可寻路位置，避免后续距离和邻接计算无法定位。

5. 生成 action_dir

action_dir 根据当前行和下一行的 pacmanPos 计算， 表示“从当前点走到下一点”的动作。最后一行没有下一步，动作缺失。

6. 生成 available_dir

在 action_dir 生成后，脚本读取 adjacent_map_fmri.csv 判断该动作是否是当前位置的合法方向。 当前字段是单个 bool，表示本行真实动作是否合法，不是四方向 bool 数组。

05. 计算 utility

脚本 script/05_calculate_utility.py

输入数据data/04_corrected_tile_data，常量目录 data/constant_data。
输出数据data/05_utility_data。

目的

集中完成策略 utility 的计算、修正和归一化，避免拟合阶段再重复处理 Q 值。

处理方式

1. 准备稳定行标识

输入 corrected tile data 后，脚本为每行生成 row_id。 该字段标注当前表内行号，用于动态拟合、权重修正和后续特征回指。 本阶段之后的数据不会删除或重排行，因此 row_id 可以作为稳定引用。

2. 读取地图常量

从 data/constant_data 读取邻接表和最短路径距离表。 邻接表用于判断墙方向和 tunnel 特例；距离表用于计算基于路径树的 utility。

3. 计算七类 raw Q

对每行四个方向分别计算策略 utility，生成 global_Q、local_Q、evade_blinky_Q、 evade_clyde_Q、approach_Q、energizer_Q、 no_energizer_Q。每个字段都是长度为 4 的数组，对应 left/right/up/down。

4. 修正不可走方向

根据当前位置的邻接表，将墙方向的 Q 写为 -np.inf。 这个修正只影响 Q 数组中的不可走方向，不改变任何行和非 Q 字段。

5. 处理 evade/no_energizer 的尺度

对 evade 和 no_energizer 类 Q，会先根据有限值最小偏移把可走方向平移到非负尺度， 再进入归一化；这样保留旧流程中这些策略的相对强度语义。

6. 生成 Q_norm

对每个 raw Q 生成对应 *_Q_norm。归一化只在有限可走方向上计算， -np.inf 的墙方向不参与最大值计算。若有限方向全为 0，则保持 0。

06. 动态策略拟合

脚本 script/06_dynamic_strategy_fitting.py

输入数据data/05_utility_data，邻接表 data/constant_data/adjacent_map_fmri.csv。
输出数据data/06_weight_data。

目的

根据每个 trial 中行为动作和七类策略 Q 值，拟合动态策略权重，并给出逐行预测方向和正确性。

处理方式

1. 校验拟合输入

输入必须已经包含七类 *_Q_norm、action_dir、 available_dir、DayTrial 和 row_id。 拟合阶段不再重新计算 Q，也不再重新推断动作方向。

2. 按 trial 切分行为序列

数据按 DayTrial 分组。每个 trial 内，模型使用当前行的七类 Q_norm 和实际 action_dir 去拟合能解释行为选择的策略权重。

3. 使用遗传算法搜索权重

每段数据用遗传算法搜索七维策略权重。默认种群大小为 100，迭代 500 次， 变异概率为 0.01，权重精度为 0.001，并带有 L1 权重惩罚。 并行时可用 --segment-workers 加速单文件内部段落拟合。

4. 预测方向并判断正确性

拟合得到的权重与七类 Q_norm 加权后形成四方向分数。 分数最高方向写入 predict_dir，与真实 action_dir 比较后写入 prediction_correct。

5. 标注 trial 上下文

输出保留 trial_context、eat_energizer、eat_ghost。 连续停留达到 stay_length 的片段标为 is_stay； 单策略正确率低于阈值时标为 is_vague。

07. 修正人类策略权重

脚本 script/07_revise_human_weight.py

输入数据data/06_weight_data。
输出数据data/07_corrected_weight_data。

目的

根据 energizer、ghost、scared 时间窗口和 stay/vague 等规则，对拟合后的归一化权重和预测结果进行修正。

处理方式

1. 按 trial 读取拟合结果

输入是动态拟合后的逐行数据。脚本按 DayTrial 分组，逐 trial 应用修正规则， 并保持原行数和行顺序。

2. 根据事件上下文修正权重

规则会检查 energizer、ghost、scared 时间窗口、stay 和 vague 等上下文。 满足条件的片段会重算策略归属，使策略解释更符合行为事件。

3. 生成修正后的字段

revised_normalized_weight 表示修正后的归一化策略权重； revised_prediction_correct 表示基于修正权重重新判断的预测正确性。

4. 生成策略编号

strategy 从修正后的权重和上下文中得到，是后续离散特征、fMRI formed 数据和 grammar 使用的核心策略标签。当前策略集合对应两鬼七策略加 stay/vague。

08. 提取人类行为特征

脚本 script/08_extract_features_human.py

输入数据data/07_corrected_weight_data，常量目录 data/constant_data。
输出数据data/08_feature_data 和 data/08_discrete_feature_data。

目的

从修正权重数据中提取连续特征和离散状态特征，为 fMRI 格式化、状态依赖图和 grammar 提供统一状态表达。

处理方式

1. 读取地图距离常量

脚本读取邻接表和最短路径距离表，用于计算 Pacman 朝四个方向移动后到 ghost、energizer 和 bean 的距离。

2. 生成连续特征

对每行计算 PG1、PG2、PE、bean 相关距离特征等。 ghost 特殊 home 坐标会映射到可寻路位置；墙方向距离使用固定大值编码。

3. 生成离散状态特征

连续距离会被离散化为状态列： PG1、PG2、PE、BW10、BB10、 IS1、IS2。其中 BW10 表示 10 步内是否有 bean， BB10 表示 10 步外是否有 bean。

4. 附加分析上下文字段

两个输出表都会保留或追加 revised_normalized_weight、 normalized_weight、weight、row_id、strategy、 DayTrial、frame_id、game_id 和 action_dir。

5. 分别写出连续表和离散表

feature_data 保留连续值，适合检查距离和权重； discrete_feature_data 只保留后续 fMRI/grammar 需要的离散状态和上下文字段。

09. human fMRI 数据预处理

脚本 script/09_human_fmri_data_preprocess.py

输入数据data/08_discrete_feature_data。
输出数据data/09_fmri_discrete_feature_data_ghost2、data/09_fmri_formed_data_ghost2、data/09_strategy_sequence。

目的

将离散特征整理成 ghost2 分析格式，并按相邻被试集合生成 grammar 所需的策略序列结构。

处理方式

1. 整理 ghost2 离散特征

当前主流程已经只保留 two-ghost 数据，因此本阶段不再做二鬼/四鬼分流。 脚本将 data/08_discrete_feature_data 中的文件写入 09_fmri_discrete_feature_data_ghost2，并确保存在 game_id。

2. 展开策略 one-hot

strategy 编号会被转换为策略列： global、local、evade_blinky、evade_clyde、 approach、energizer、no_energizer、 vague 和 stay。列值使用旧流程的 1/2 编码。

3. 提取连续策略段首点

在每个 game_id 内，如果当前行策略 one-hot 与上一行相同，就视为同一连续策略段； formed 数据只保留每个连续策略段的第一行。这样 grammar 学到的是策略切换序列，而不是逐帧重复状态。

4. 生成 strategy sequence

脚本使用近邻关系把被试组成分析集合，并为每个集合输出： seq（策略符号序列）、S（状态矩阵）、 state（状态名）、strategy（策略列名）、 strategyLabel（策略符号）和 fileNames（来源文件）。

5. 删除不再使用的聚合字段

当前流程保留 evade_blinky 和 evade_clyde，不再生成聚合 evade 字段，避免后续状态表混入未使用维度。

10. 生成状态依赖图

脚本 script/10_state_dependency_graph.py

输入数据data/09_strategy_sequence。
输出数据data/10_state_dependency_graph_data。

目的

根据策略序列中的状态变量，估计状态之间的依赖关系，为 grammar 生成提供结构约束。

处理方式

1. 读取策略序列对象

每个输入 pickle 都来自 09_strategy_sequence，包含策略符号序列和对应状态矩阵。 文件名保持被试 id，输出也按同名文件写出。

2. 选择状态变量

默认参与依赖图计算的状态是 IS1、IS2、PG1、PG2、PE、 BW10。BB10 保留在上游数据中，但当前状态图默认不使用。

3. 估计状态依赖关系

脚本根据状态矩阵中的状态共现关系计算状态之间的依赖强度，并使用 alpha 作为 Dirichlet 先验平滑强度，避免稀疏状态导致概率为零。

4. 写出每个被试的图结构

每个输出 pickle 保存该被试对应的状态依赖图数据。下一阶段 grammar 学习会同时读取该图和对应策略序列。

11. 生成 grammar

脚本 script/11_generate_grammar.py

输入数据data/09_strategy_sequence 和 data/10_state_dependency_graph_data。
输出数据data/11_grammar。

目的

基于策略序列和状态依赖图，学习每个被试的 grammar chunk、解析结果和 skip-gram 信息。

处理方式

1. 配对读取序列和状态图

对每个被试文件，脚本读取 09_strategy_sequence 中的策略序列， 以及 10_state_dependency_graph_data 中同名状态图。两者共同决定 chunk 候选的统计和约束。

2. 初始化基础 grammar

初始 token 来自策略符号，例如 G、L、1、2、 A、E、N、S、V。 参数中记录状态名、学习系数、最大迭代次数、收敛窗口和候选保留阈值。

3. 迭代学习 chunk 候选

学习过程根据策略序列中相邻 token 的组合频率、状态依赖约束和候选得分生成 chunk。 达到收敛条件或最大迭代次数后停止。--quiet 只影响日志，不影响结果。

4. 解析序列并保存结果

输出对象包含 grammar、parsed 和 skip_gram。 grammar 中 components 第二项非空的条目就是实际学到的 chunk， 例如 E-A=E+A 或 E-A-G-L=E-A+G-L。

5. 记录来源和参数

source 保存输入文件和参与者集合，parameters 保存学习参数。 这些元数据用于解释结果来源，不参与下游数值计算。

12. DividePerson 人群划分

脚本 script/12_divide_person.py

输入数据data/11_grammar。
输出数据不保存文件，只在命令行打印 JSON。

目的

根据每个被试 grammar 中不同类型 chunk 的比例，将被试划分为指定数量的群体。 当前默认聚类数量为 2。

处理方式

1. 读取 grammar 结果

脚本读取 data/11_grammar 下所有被试文件。每个文件中的 grammar 列表会被解析为该被试的 chunk 集合。

2. 构造聚类特征

对每个被试统计不同 chunk 类型的比例，形成低维特征向量。 当前默认使用两个聚类，沿用旧 DividePerson 的二分逻辑。

3. 聚类并整理每组 grammar book

输出会列出每个被试的 label，并按 label 汇总该组出现的 grammar token 和组件关系。 这一步用于解释人群层面的策略结构差异。

4. 只打印不落盘

该脚本是后处理检查工具，结果以 JSON 打印到命令行，不写入 data。 若需要保存，可在运行时由 shell 重定向输出。

视频流程

视频流程不参与主分析结果生成，主要用于把 frame data、grammar 标注和渲染配置转换成图片帧与视频。 默认 subject 为 041122-403，可以通过位置参数指定其它被试。

视频 01. 生成 render table

脚本 script/pacman_video/run_render_table.py

输入数据data/02_frame_data 和 data/pacman_video/grammar_data。
输出数据data/pacman_video/render_data。

目的

将原始 frame data 和视频用 grammar 标注合并为渲染表，作为逐帧绘图的数据输入。

处理方式

1. 定位被试数据

脚本根据 subject 参数在 data/02_frame_data 中查找对应 raw frame data。 视频流程使用 raw frame data，是因为渲染需要像素坐标、方向帧、水奖励等主分析预处理后已删除的字段。

2. 读取视频 grammar 标注

从 data/pacman_video/grammar_data 读取用于展示的 grammar 或策略标注， 并与 frame 行按 trial/frame 信息对齐。

3. 筛选 ghost trial 类型

--ghost-count 控制渲染两鬼、四鬼或全部 trial。 该参数只影响视频展示，不改变主分析流程的数据假设。

4. 写出渲染表

输出 render table 到 data/pacman_video/render_data。 render table 是图片帧绘制的唯一输入，包含地图状态、角色位置和侧边栏展示信息。

视频 02. 渲染图片帧

脚本 script/pacman_video/run_frame_renderer.py

输入数据data/pacman_video/render_data。
输出数据data/pacman_video/frame_images。

目的

把 render table 中的逐帧状态绘制为图片序列，用于人工检查或后续合成视频。

处理方式

1. 读取 render rows

脚本从 data/pacman_video/render_data 读取指定 subject 的渲染表。 每一行描述一帧画面所需的地图、角色和附加标注。

2. 选择渲染范围

--trial 可限制某个 trial；--start 指定起始帧； --max-frames 限制最多渲染帧数。调试时通常先小范围渲染。

3. 绘制单帧图像

渲染器根据 frame 状态绘制迷宫、Pacman、两只或四只 ghost、豆子、energizer 和侧边栏。 --bar-type 控制侧边栏内容，默认显示 grammar。

4. 写出 PNG 序列

图片帧写入 data/pacman_video/frame_images/<subject> 下的分 trial/game 目录。 文件名按帧序号排列，供视频合成脚本直接读取。

视频 03. 合成视频

脚本 script/pacman_video/run_video_renderer.py

输入数据data/pacman_video/frame_images。
输出数据data/pacman_video/video_data。

目的

将逐帧图片合成为 mp4 视频，方便播放和展示 Pacman 行为过程。

处理方式

1. 扫描图片帧目录

脚本读取 data/pacman_video/frame_images/<subject> 下的各个 game/trial 图片目录。 每个目录会独立合成为一个视频。

2. 校验输出策略

默认不会覆盖已有 mp4；只有传入 --overwrite 时才会重新写出。 输出目录为 data/pacman_video/video_data/<subject>。

3. 调用 ffmpeg 编码

图片序列按文件名顺序送入 ffmpeg。默认帧率为 30 fps； --crf 控制质量，数值越低质量越高；--preset 控制编码速度和压缩效率。

4. 输出 mp4 和日志

每个图片帧目录输出一个 .mp4，命令行打印状态、game 名称、帧数和输出路径， 便于检查是否有空目录或跳过的已存在视频。
